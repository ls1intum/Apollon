# Apollon — Technical and Organizational Measures (TOMs)

Documents the measures taken per Art. 32 GDPR. Structured along the categories of BayLfD / DSK TOM-Leitfäden.

*Stand: 2026-04-24 · Version 1.0*

## 1. Confidentiality (Art. 32(1)(b))

### 1.1 Access control (Zutrittskontrolle)

- Apollon runs on AET-managed servers hosted in a TUM server-room environment (power, cooling, and fire-detection provided by TUM / AET facility management).
- Physical access is restricted to AET staff and authorised TUM facility personnel.

### 1.2 System access (Zugangskontrolle)

- Host VMs are reachable over SSH only, using SSH key pairs; password authentication is disabled.
- The GitHub deployment pipeline uses `VM_SSH_PRIVATE_KEY` held as a per-environment GitHub secret, rotated on demand.
- The Apollon webapp itself has **no authentication** — every user is anonymous — so there is no user-facing access-control surface to administer.

### 1.3 Data access (Zugriffskontrolle)

- Only AET operators with container-host SSH access can read Redis data directly. Redis is bound to the internal Docker network; no external port is exposed (see `docker/compose.db.yml` — `expose: ["6379"]`, no `ports:` mapping).
- Traefik exposes only `/`, `/api`, and `/ws` externally. All other paths return 404.
- The server accepts anonymous POST/GET on `/api/...`. Authorisation to a specific diagram is enforced by possession of its unguessable share-link ID. Diagram IDs are UUIDv4 (~122 bits of entropy), generated by the `uuid` package in `library/lib/utils/index.ts`; a brute-force guess is practically impossible.

### 1.4 Separation (Trennungskontrolle)

- Redis data is not co-located with other TUM services.
- Diagrams are keyed by an opaque diagram ID; no cross-service or cross-user linkage is performed.

## 2. Integrity (Art. 32(1)(b))

### 2.1 Transport integrity (Weitergabekontrolle)

- All external traffic is TLS-terminated at Traefik using publicly trusted certificates (Let's Encrypt). HTTP is redirected to HTTPS with HSTS `max-age=63072000; includeSubdomains; preload`.
- WebSocket traffic uses `wss://` (TLS) in production; the relay forwards messages in memory only.
- Internal traffic between the webapp, server, and Redis flows exclusively over the internal Docker bridge on the same host.

### 2.2 Input control (Eingabekontrolle)

- Git is the authoritative source for all application code; changes are traceable via signed commits and pull-request history.
- The GitHub Actions deployment pipeline pushes images tagged by the source commit SHA; cosign-signed images are the only artifacts allowed in production.

## 3. Availability and resilience (Art. 32(1)(b))

- Containers restart automatically on failure (`restart: unless-stopped`).
- Docker health checks run every 5–30 s per service; unhealthy containers are marked and restarted.
- A maintenance page (served by nginx) is configured to appear when the application compose is down.
- **Backups:** No backups containing personal data exist. Source code (Git), configuration, and container images (GHCR) are backed up as part of the standard development lifecycle, but these contain no user data. Restoration from cold state reproduces the service topology without restoring any personal data.

## 4. Review and evaluation (Art. 32(1)(d))

- Privacy page and imprint source are versioned in the repository under `standalone/webapp/public/legal/profiles/tumaet/`; every change is reviewable in pull requests.
- End-to-end tests (Playwright) cover the public pages, including legal pages and hostile-markdown sanitisation.
- Dependency scanning runs on CI (npm audit + Dependabot).
- Docker images are signed with cosign and pushed to GHCR; image provenance is verifiable.

## 5. Pseudonymisation / encryption

- Diagrams have no linkage to an identifying user; the diagram ID itself is a cryptographically random pseudonym.
- Network traffic is TLS-encrypted end-to-end (browser → Traefik → internal services).
- Redis data at rest is not separately encrypted by the application. Compensating controls: Redis is bound to the internal Docker bridge only; no external port is exposed; Redis contains only user-authored diagram content with a 120-day native TTL; host-level disk protection follows the AET operational baseline. If the DSB requests application-level encryption at rest, options include a LUKS-encrypted Docker volume for `dbdata` or Redis Enterprise at-rest encryption — tracked as a roadmap item rather than a current control.

## 6. Deletion and data minimisation at source

### 6.1 Diagram content

Redis TTL of 120 days is enforced natively by the Redis engine (`EX` parameter on `SET`, configured in `standalone/server/src/database/models/Diagram.ts` via `DIAGRAM_TTL_SECONDS = 120 * 24 * 3600`). No scheduled job is required; no external log shipper or host-level tooling is involved.

### 6.2 Per-request access logs

Not produced anywhere in the stack. `nginx.conf` sets `access_log off;`. `docker/compose.proxy.yml` sets `--accesslog=false` on Traefik. The Express server logger runs at level `silent` in production (`standalone/server/src/logger.ts`). No persistent record of client IP, URL, or user-agent is created in the course of normal operation.

### 6.3 Operational events (no personal data by design)

All remaining log output — service startups, Let's Encrypt renewals, Traefik service-level errors, Redis engine events, and crit-level nginx errors — concerns the service itself, not end users. To prevent nginx's default error log format (which includes the client IP) from emitting personal data, `nginx.conf` sets `error_log /dev/stderr crit;`. Because no personal data is retained, Art. 5 Abs. 1 lit. e DSGVO imposes no time-based retention requirement for this stream. The Docker `json-file` log driver (`max-size=50m`, `max-file=5`) supplies a size-bounded ring buffer (~250 MB per container) as defense-in-depth and to preserve operator debugging capability. The entire configuration is versioned in `docker/compose.*.yml` and `nginx.conf` and deploys through the existing GitHub Actions workflow; no host-level log shipper, cron job, or `/etc/logrotate.d/` drop-in is required.

### 6.4 Incident fallback

If a log line is ever observed to contain personal data — for example, after a future code or configuration change inadvertently re-enables per-request logging — the event is handled as a personal-data breach under Art. 33 DSGVO: the TUM Data Protection Officer is notified within 72 hours, the affected log segment is cleared, and the underlying code or configuration is corrected to prevent recurrence. See §7.

## 7. Organisational measures

- Operators are TUM / AET employees or authorised contributors acting under Art. 4 BayHIG and TUM-internal security policies.
- The TUM Data Protection Officer (<beauftragter@datenschutz.tum.de>) reviews incidents affecting personal data. Data-subject requests follow the process in `docs/admin/legal-pages.md` and on the live `/privacy` page.
- Security incidents are recorded in GitHub Issues against the `ls1intum/Apollon` repository. Incidents that affect personal data — including any discovery of personal data in operational logs (see §6.4) — are reported to the TUM DPO within 72 hours in accordance with Art. 33 GDPR.

## 8. Sub-processors

None. All components run on TUM / AET infrastructure.
